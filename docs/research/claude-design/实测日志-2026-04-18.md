## Phase 1：bundle 解包与理解

### 1.1 Bundle 协议的三层 payload

解包后得到 3 类文件，对 coding agent 的价值完全不同：

| 文件 | 角色 | 阅读顺序 | 不读的代价 |
|------|------|---------|-----------|
| `README.md` | **协议指令书**（写给 coding agent） | 第一读 | 不知道"匹配视觉 ≠ 复制结构"、会被诱导复制 HTML |
| `chats/chat1.md` | **意图记录**（用户的约束、选项、澄清） | 第二读 | 只读 HTML 会错过"不要四个一样大卡片"这种硬约束 |
| `project/*.html` | **最终输出**（pixel-perfect 的视觉+交互规范） | 第三读，读全 | skim 会丢失 CSS tokens、交互脚本、数据模式 |

**摩擦 #1**：README 显式禁止"在浏览器渲染或截图"—— 要求直接读 HTML/CSS 源码。这是违反我"先看再说"的默认习惯的。如果没 README 明说，我大概率会先跑一次截图。

### 1.2 意图 vs 输出的差

chat1.md 记录了用户对 assistant 的 questions_v2 回答：
- `hierarchy_logic: 混合流 —— 以『今日最热』跨赛道单品为主视觉`
- `data_density: 中等——首屏 15-20 条`
- `hero_element: 要——跨赛道精选 1 条，超大`
- 后续追加："加一个颜色清爽点的主题"→ 产生 `fresh` 第三档主题

这些**在 HTML 里能推断但不确凿**的信息，只有 chat 能给出权威来源。例如 HTML 里有 Hero、有三主题切换，但"为什么三档"、"fresh 是用户追加的"、"data_density 选了中等"—— 只有 chat 有。

**摩擦 #2**：如果 L3 卡只写"读 README + HTML"，会漏掉 chat 这一层。协议实际是三层：**协议 / 意图 / 输出**。

## Phase 2：基线代码读

### 2.1 可复用 vs 必新建（/trending 现状）

| 维度 | 现状 | /trending-v2 |
|------|------|--------------|
| 数据层 `fetchTrendingData(days)` | 完整提供 papers/tools/articles/signals/health/timestamps | 100% 复用，零修改 |
| ISR 机制 | `export const revalidate = 300` | 同样 |
| Async params (Next 15) | 已处理 | 同样 |
| 布局 | 2×2 grid 并列 4 TrackSection | 单列叙事流 + 360px sticky aside —— **全新** |
| Hero（跨赛道 Top 1） | 无此概念 | 新建 + 需新算法 |
| Aside（Momentum / Top 5 / 关键词云） | 无 | 新建 |
| 主题 | next-themes 二档 light/dark（class） | 三档 light/fresh/dark（data-theme）—— 机制不兼容 |
| 字体 | Inter + Geist Mono | + Noto Serif SC / Fraunces italic / JetBrains Mono |

### 2.2 关键张力：token 系统粒度

- SkillNav 现有 tokens：粗粒度 semantic（`--background`、`--muted-foreground`、`--border`）
- bundle tokens：细粒度物化（`--ink / --ink-2 / --ink-3 / --ink-4`、`--line / --line-2`、`--accent-ink`）

两者**不能直接对应**。如果蒸馏到 SkillNav 现有 semantic，会塌陷层次；如果全量引入 bundle tokens，会造成双系统并存。

**决策（用户 D1 = C 折中）**：
1. **产品级 token 升全局**：赛道色 `--color-track-paper/tool/news/community` 入 `globals.css`（长期复用）
2. **视觉细粒度 scope 在 v2**：细粒度 ink/line vars 只在 `/trending-v2` scope 内定义，不污染全局
3. **字体升全局**：Noto Serif SC / Fraunces 作为全局字体变量，其他未来页面也能用

| 领域 | bundle 要求 | SkillNav 映射 | 决策类型 |

| **CSS vars 墨色** | `--ink / --ink-2 / --ink-3 / --ink-4` | v2 scope 内定义 | scope 在 v2 |
| **CSS vars 线条** | `--line / --line-2` | v2 scope 内定义 | scope 在 v2 |
| **强调色** | `--accent` / `--accent-ink`（三主题各不同） | v2 scope 内定义 | scope 在 v2 |
| **赛道色** | `--gold` / `--green` / `--violet` / `--blue` | 升全局为产品 token `--color-track-*` | 升全局 |
| **响应式** | 1100px 单列降级 | 自定义 media query `@media (max-width: 1100px)` 在 v2.css（不用 Tailwind 断点） | scope 在 v2 |
| **Hero Top 1 数据** | 编辑精选 + 热度 9,847 ↑312% | 归一化算法（papers upvotes / tools delta / articles relevance / signals score 各自 max 归一化）选最高 | 新增数据层函数 |
| **Tweaks panel** | postMessage 协议 + 右下悬浮面板 | **省略**（Tweaks 是 Claude Design 预览环境的协议，production 不需要） | 不实现 |
| **搜索框** | ⌘K placeholder | **占位符**（不接 Orama），UI 占位 | 降级 |
| **布局容器** | `max-width: 1440px` + `padding: 0 40px` | 用 Tailwind `max-w-[1440px]` + `px-10` | 直接映射 |
| **sticky aside** | `position: sticky; top: 88px` | 原样 | 直接映射 |

### 3.1 关键降级（production vs prototype）

1. **Tweaks panel 不做**：Claude Design 预览协议，production 无意义
2. **搜索框不接真后端**：占位 UI，保持视觉一致
3. **浏览量 "12,847"**：mock 或隐藏（SkillNav 没接 Umami real-time API）
4. **"Figure 3 · 自博弈收敛曲线" caption**：用真论文的 HF API `thumbnail` 或 org + date 做动态文案，不 mock
5. **"作者 K. Amodei 等 12 人"**：真论文有 authors，mock 数据没有，动态渲染

**摩擦 #3**：bundle 充满"prototype 数据"（mock 内容）。coding agent 必须判断**哪些是视觉要求（要 pixel-perfect）、哪些是数据模式（要替换）**。README 说"匹配视觉不复制结构"是给结构的规则，但"数据 vs 视觉"的边界 README 没显式说。

## Phase 4：实现摩擦点

### 4.1 文件尺寸 vs 设计密度

CLAUDE.md 规定"单文件不超 300 行，超限拆分"。这次两处违规：
- `trending-v2.css` **540 行** —— bundle 是视觉密集场景，scoped CSS 自然膨胀
- `main-feed.tsx` **~400 行** —— 4 赛道 × (BigItem + CompactRow) = 8 小组件

**判断**：硬拆 CSS 会破坏阅读性（token 和 component 样式穿插才好查）；硬拆 feed 组件也会产生过度碎片化（每个 track 一个文件，但它们共享 helpers）。

**张力本质**：通用编码规范（300 行）是为"业务逻辑密度"设定的，遇到"视觉密度"就偏紧。这是 L3 候选之一：**密度不同的代码应有不同的行数阈值**，或"密度 = 视觉 + 配置"的 file 豁免。

### 4.2 ThemeShell 反模式 → useSyncExternalStore

第一版用 `useState + useEffect(() => setState(readLocalStorage()))` 被 ESLint 规则 `react-hooks/set-state-in-effect` 拦。重写为 `useSyncExternalStore(subscribe, readTheme, serverSnapshot)`，避开 SSR hydration flash，且干净通过。

**收获**：项目级 ESLint 规则推动你走向更优解。`.claude/rules/react-nextjs.md` 里已经明记 "use `useSyncExternalStore`"，这是一条**规则即硬约束**的典型示例 —— rules 子目录的存在让 coding agent 不用重新发明。

### 4.3 "数据 vs 视觉"的边界模糊

bundle mock 了 "热度指数 9,847 ↑ 312%"、"作者 K. Amodei 等 12 人" 这类字段，真实数据没有。

决策：**可计算的用真数据（upvotes、weekly_stars_delta、relevance_score），不可计算的静默降级**（去掉作者数、去掉 delta 百分比）。

**摩擦**：README 没告诉我"数据 mock 和视觉 mock 的边界"。视觉 mock（颜色、字号、间距）必须 pixel-perfect；数据 mock（具体数字）必须替换。这个区分是**默认常识但 README 没显式说**。

### 4.4 "跨赛道 Top 1" 算法

bundle 是编辑精选（静态 mock）。生产必须有算法。我实现了"各赛道 max 归一化 + 全集取最高"，最低成本可用。

**问题**：四赛道分数单位不同（upvotes vs weekly_stars_delta vs relevance_score vs HN score），归一化只是掩盖了不可比性。Hero 经常会被单个赛道长期霸占。真要做好需要**编辑标记 + 人工 override**（数据层加 featured bool）。

这是典型的**新功能的"现有数据够不够用"判断**，coding agent 容易过度相信数据层够用。

### 4.5 Bundle 的哪些条款实际用上 / 没用上

用上的：
- ✅ "先读 chat" —— 捕获到 data_density、三主题由来
- ✅ "不复制 HTML 结构" —— 我没保留 `.masthead`、`.hero` 这类无前缀类名，全部 `v-` 前缀避免全局碰撞
- ✅ "模糊问用户" —— 3 个 Open Decisions 都先问
- ✅ "读完 project/*.html 不 skim" —— 最后 `.tweaks` panel + JS 片段不读会错过 postMessage 协议（我主动降级）

没用上的：
- ❌ "跟随 imports" —— bundle 只有 1 个 HTML，无 imports
- ❌ "如果渲染需要截图先问用户" —— 我没渲染 bundle，直接读源码

### 4.6 产物分类的实战信号（4 子格假设验证）

Vault HANDOFF 预设的"4 子格 = 提示词 / Skill / Agent / Harness"。本次实测产出：

| 产物 | 性质 | 4 子格归属？ |
|------|------|-------------|
| `docs/research/claude-design/实测日志-2026-04-18.md` | 一次性实测记录 | **都不属于** — 是"实证材料" |
| `docs/research/2026-04-18-claude-design-usage-guide.md`（上次会话） | 使用指南草稿 | 提示词？但它更像 SOP/Runbook |
| 即将写的 `Claude-Design-L3-1-handoff协议.md` | 抽象协议知识 | **都不属于** — 是"知识卡/L3"，这是 Vault 既有层级 |
| `/trending-v2/*.tsx + .css` 代码 | 工程脚手架 | Harness？但它更像"项目代码"而非 harness |
| ESLint 规则 `react-hooks/set-state-in-effect` | 规则即约束 | **都不属于** — 是"项目级规约" |
| `.claude/rules/react-nextjs.md` | 规则文档 | 提示词？但它是静态文档不是 prompt |

**初步观察**：4 子格**至少漏了两类**——"实证材料/日志"、"项目级规约/rules"。且"提示词"这个格模糊（SOP、使用指南、静态规则都是"写给 AI 读的文本"但性质不同）。

**这个信号必须落到 L3 卡里**：Step 3 评估宪法时，这是重要输入。
