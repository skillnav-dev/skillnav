## 目标（为什么做这件事）

提高人机协同效率，4 维度可观测：

| 维度 | 含义 |
|------|------|
| 启动效率 | 想做→AI 能开工的时间 |
| 对齐效率 | 过程中消除误解的成本 |
| 沉淀效率 | 做完→形成可复用资产的成本 |
| 激活效率 | 下次相似任务时旧资产自动召回 |

**长期动机**：后续要搭 RAG / 可视化 / 内容管线等上层组件。底座越坚实，上层越容易搭。

## 第一轮评估 · 3 agent 并行（2026-04-18）

### A1 · 最优解调研 + 框架候选

3 候选：
1. **Claude Code 原生四件套**（Skill/Hook/Subagent/MCP）—— 同构性最高，SkillNav 现状匹配
2. **LangGraph + AutoGen/CrewAI** —— 扩展性好，但单人场景复杂度溢出
3. **Norman 双鸿沟 × OS 分层** —— 人机覆盖最好，但抽象偏静态

**关键洞察**：v0.2 图 A 的 "Runtime 7 模块" ≈ 候选 1 的二手变体，应直接用候选 1。

### A2 · 骨架批判 + 底座视角

**6 个结构问题**：
- C1 Hook 错层（图 A / 图 B 双重身份）
- C2 "对齐层 vs 触发层" MECE 违反（认知契约 vs 执行单元混层）
- C3 Guardrail（运行时）vs 约束层（声明文件）语义重叠
- C4 Context / Memory 被降格为旁路（应是一等公民）
- C5 L3→L2 反哺未画回环
- C6 **缺度量层**——无法回答"协同效率提升了吗"

**5 个底座硬缺口**：
- 扩展点（RAG/可视化/管线接哪）
- 接口契约（input/output/error/timeout 四元组）
- 版本治理
- 运行时注册表（不是静态清单）
- 能力发现机制（capability manifest）

### A3 · 反方 3 刀

- **过度设计**：8 种载体实际只有 3 类（可执行单元 / 上下文源 / 触发机制）
- **底座伪需求**：SkillNav 跑到今天靠 CLAUDE.md + MEMORY.md + 几个 script，不需要底座
- **Runtime 让人被架空**：协同瓶颈从来是"人说不清要什么"，不是"AI 执行不顺"
- **文档宿命三缺一**：缺触发场景 / 缺降级路径 / 缺回收机制——下一份坟场文档
- **双图精神分裂**：图 A 图 B 的 Context 不是同一个东西
- **N=1 过拟合**：至少跑 3 次（RAG / 可视化 / 内容管线各一次）再提骨架
