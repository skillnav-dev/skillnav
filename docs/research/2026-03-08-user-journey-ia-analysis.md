# 用户旅程与信息架构分析

**分析日期**: 2026-03-08
**分析依据**: 项目代码审查 + 内容战略 v2 + 竞品分析 + 市场趋势

---

## 分析概要

SkillNav 当前站点是一个典型的"三合一工具站"结构：Skills 导航 + 文章资讯 + MCP 导航，外加关于页。站点在技术实现上成熟（搜索/分类/分页/SEO 完备），但信息架构存在三个核心矛盾：

1. **导航心智模型冲突** — "Skills / MCP / 资讯" 三个板块按产品形态而非用户意图组织，用户需要先理解 "Skill 和 MCP 有什么区别" 才能导航，这对目标用户（中文开发者）不友好。
2. **内容价值层次未分** — 翻译文章和即将推出的编辑精选（周刊/评测）混在同一个 /articles 路由下，无法让用户快速感知"这个站有自己的观点"。
3. **三个板块孤岛化** — Skill 详情页无关联文章，文章详情页无关联工具，MCP 页面完全静态独立，无法形成内容间的交叉发现。

本报告从用户画像出发，映射关键旅程，识别流失点，并提出信息架构重构方案。

## 1. 用户画像

### 画像 A: 实战开发者 — "效率猎手"

| 维度 | 描述 |
|------|------|
| **角色** | 全栈/后端工程师，2-5 年经验，日常使用 Claude Code 或 Cursor |
| **技术水平** | 中高，能独立配置 CLI 工具和 MCP Server |
| **使用场景** | 正在做项目，想找一个能加速特定任务的 Skill（如代码审查、测试生成） |
| **核心需求** | 快速找到、评估、安装一个 Skill，5 分钟内完成 |
| **行为模式** | 搜索引擎直达详情页 > 快速扫描安装命令和质量评级 > 安装或离开 |
| **停留时长** | 首次 2-5 分钟（找工具），回访 1-2 分钟（直奔目标） |
| **回访频率** | 有需求时回访（月 1-3 次），如果订阅周刊可能变为周访 |
| **转化目标** | 安装 Skill → 订阅周刊 → 分享给同事 |

**关键洞察**: 这类用户是站点的"核心交易用户"，他们带着明确意图来，对搜索质量和安装体验极度敏感。当前 Skills 列表的搜索+分类已能满足基本需求，但缺少"编辑推荐理由"和"同类对比"来帮助决策。

### 画像 C: 探索新手 — "好奇观望者"

| 维度 | 描述 |
|------|------|

## 3. 信息架构重构方案

### 当前架构评估

**当前导航**: 首页 | Skills | MCP | 资讯 | 关于

| 问题 | 严重程度 | 说明 |
|------|---------|------|
| Skills 和 MCP 并列导致认知负担 | 高 | 用户需要先理解两者区别才能决策点哪个 |
| "资讯"命名太泛 | 中 | 不暗示内容质量或编辑深度，和新闻聚合站无差异 |
| MCP 页面只有 18 个静态卡片 | 中 | 作为独立导航项，内容太薄，第一次点击后不会再回来 |
| 无全站搜索 | 中 | 用户如果搜"code review"，不确定该去 Skills 搜还是资讯搜 |
| 首页无"首页"导航项实际不需要 | 低 | Logo 已可回首页，但当前 navItems 包含"首页" |

### 推荐架构

**新导航结构**: 工具 | 内容 | 关于

展开为：

```
工具 (dropdown)
  ├── Skills 精选     → /skills          (168 精选 Skills)
  ├── MCP 导航       → /mcp             (精选 MCP Server)
  └── 工具对比       → /compare          (未来: 同类工具横评)

内容 (dropdown)
  ├── 周刊           → /weekly           (SkillNav Weekly)
  ├── 深度文章       → /articles?tier=editorial
  └── 翻译资讯       → /articles?tier=translated

关于                → /about
```

**替代方案（更简洁，推荐 MVP 阶段采用）**:

```
探索      → /skills      (Skills + MCP 合并展示，Tab 切换)
周刊      → /weekly       (品牌核心内容，独立入口)
资讯      → /articles     (全部文章，Tab: 编辑精选/翻译资讯)
关于      → /about
```

**推荐采用替代方案的理由**:
1. 导航项不超过 4 个（移动端友好）
2. "周刊" 独立出来是内容战略 v2 的核心需要——它是品牌载体
3. "探索" 统一 Skills + MCP，降低用户认知成本
4. 去掉"首页"导航项（Logo 已承担此功能）

### URL 结构优化

| 当前 | 推荐 | 理由 |
|------|------|------|
| `/skills` | `/skills` (保持) | Skills 是核心 SEO 关键词，不改 |
| `/skills/[slug]` | `/skills/[slug]` (保持) | 已有 SEO 积累 |
| `/mcp` | `/mcp` (保持) | 少量静态页，保持即可 |
| `/articles` | `/articles` (保持) | 翻译文章聚合，保持 |
| `/articles/[slug]` | `/articles/[slug]` (保持) | 已有 SEO 积累 |
| (不存在) | **`/weekly`** | **新增**: 周刊列表页 |
| (不存在) | `/weekly/[slug]` | **新增**: 周刊详情页，slug 格式 `2026-w11` |
| `/about` | `/about` (保持) | 保持 |

**关键决策: 周刊用独立 URL 还是复用 /articles**

推荐方案: **独立 `/weekly` 路由，数据仍存 articles 表**

理由:
- 周刊是品牌载体，需要独立 SEO 权重（"/weekly" 路径让搜索引擎理解这是系列内容）
- 周刊详情页需要专属模板（目录导航、往期切换、分享按钮），和普通文章 layout 差异大
- 数据层复用 articles 表 + `series = 'weekly'` 查询，无需新建表
- 用户搜索 "skillnav weekly" 或 "AI 工具周刊" 时，独立路由有更好的 SEO 信号

### 内容分类体系重构

**Skills 分类（保持不变）**: 当前 10 个场景分类（编码开发/AI 智能体/数据处理...）是按使用场景组织的，符合画像 A 的心智模型。

**Articles 分类（微调）**:

| 当前 article_type | 含义 | 推荐调整 |
|-------------------|------|---------|
| tutorial | 教你做 | 保持，增加子标签"入门"/"进阶" |
| analysis | 帮你想 | 保持 |
| guide | 帮你选 | 保持 |

真正需要的不是改分类，而是**增加维度**。content_tier (editorial/translated) + series (weekly/monthly-roundup) 已在内容战略 v2 中设计，这是正确的方向。

**MCP 分类（保持不变）**: 当前静态数据中的分类（开发工具/数据库/搜索引擎...）合理。

### 跨内容关联设计

这是当前架构最大的短板。三个板块完全孤岛化。

**推荐关联机制**:

| 关联方向 | 实现方式 | 优先级 |
|---------|---------|--------|
| Skill → 相关文章 | 基于 Skill 名称/分类匹配 articles 表 | P1 |
| Article → 提到的工具 | 文章中手动标注 `mentioned_skills[]` 字段，或基于关键词匹配 | P1 |
| Article → 相关 MCP | 文章中手动标注或关键词匹配 | P2 |
| Skill ↔ MCP | MCP 可能是 Skill 的依赖，通过 `requires_bins` 关联 | P3 |
| 周刊 → Skills + Articles | 周刊内容中嵌入链接（Markdown 内链） | P0（随周刊发布自然实现） |

**最小可行方案**: 在周刊 Markdown 中使用站内链接 (`/skills/xxx`, `/articles/xxx`)，渲染时自动识别为站内链接并添加样式。这不需要 DB 改动，只需前端渲染层的优化。

### 搜索策略

**当前**: 每个列表页（Skills, Articles, MCP）各自独立搜索，MCP 是客户端搜索。

**推荐**: 保持分区搜索 + 增加全站搜索入口。

| 方案 | 说明 | 优先级 |
|------|------|--------|
| 保持 Skills 列表搜索 | nuqs + Supabase 查询，已运行良好 | 维持现状 |
| 保持 Articles 列表搜索 | 同上 | 维持现状 |
| **增加 Header 全站搜索** | Cmd+K 弹窗，同时搜 Skills + Articles + MCP | P2 |
| 首页增加搜索框 | Hero 区下方放一个全站搜索框 | P1 |

**首页搜索框的价值**: 对画像 A（效率猎手）而言，首页搜索框可以直接从首页开始查找工具，跳过"先选 Skills 还是 MCP"的决策。

## 4. 首页重构方案

### 当前首页模块

```
1. HeroSection — "发现最好用的 AI Agent 工具" + 2 CTA
2. StatsBar — 13,000+ / 8,500+ / 25,000+ / 12% (数据疑似虚构)
3. FeaturedSkills — 精选 Skills 卡片 (3 col grid)
4. LatestArticles — 最新资讯 (2 col grid)
5. NewsletterCta — 订阅 AI Skills 周报
```

### 首屏设计（第一屏，用户不滚动就能看到的部分）

**当前问题**:
- Hero 标题"发现最好用的 AI Agent 工具"偏工具导航叙事，不传达编辑价值
- StatsBar 数据不真实（13,000+ Skills 但只有 168 可见；25,000+ MAU 无依据）
- 两个 CTA（浏览 Skills / MCP 导航）没有覆盖"内容消费"入口

**推荐首屏方案**:

```
┌────────────────────────────────────────────────────┐
│  SkillNav — 中文开发者的 AI 工具生态指南            │
│                                                     │
│  精选信源 · 编辑策展 · 每周更新                      │
│                                                     │
│  ┌──────────────────────────────────────┐           │
│  │ 🔍 搜索 Skills、文章、MCP Server...   │           │
│  └──────────────────────────────────────┘           │
│                                                     │
│  [浏览工具]  [阅读周刊]  [最新资讯]                  │
│                                                     │
├────────────────────────────────────────────────────┤
│  168 精选 Skills  |  270+ 翻译资讯  |  13 一手信源   │
│  (链接到 /skills)   (链接到 /articles)               │
└────────────────────────────────────────────────────┘
```

**改动要点**:

3. **增加全站搜索框** — 对画像 A 直接有用，对画像 C 提供明确的"下一步"
4. **三个 CTA 覆盖三类用户** — 工具（画像 A）、周刊（画像 B）、资讯（画像 D）
5. **StatsBar 使用真实数据** — 168 Skills / 270+ 资讯 / 13 信源（或增加数据链接到来源）

## 5. 转化漏斗

### Newsletter 触点矩阵

| 位置 | 当前状态 | 推荐 | 优先级 |
|------|---------|------|--------|
| 首页底部 | 有（NewsletterCta 组件） | 保持，更新文案 | 维持 |
| 文章详情页底部 | **无** | **新增**: 文章阅读完毕后，内联 Newsletter CTA | **P0** |
| Skill 详情页侧边栏 | **无** | **新增**: "获取工具推荐周刊"小 CTA | P1 |
| 周刊详情页 | **不存在** | **新增**: "订阅下期"固定 bar | P0 |
| 文章列表页 | **无** | **新增**: 列表中间插入 Newsletter 卡片 | P2 |
| 404 页面 | 无 | 增加"订阅获取最新内容"引导 | P3 |

**最高 ROI 触点**: 文章详情页底部。原因：画像 D（渠道读者）从外部链接进入文章详情页，阅读完毕后是订阅意愿最高的时刻。当前文章详情页 (`/articles/[slug]/page.tsx`) 底部只有"相关文章"，完全没有订阅入口。

**推荐实现**: 在 ArticlePage 的 `related` section 之前，插入一个轻量级 Newsletter CTA（比首页版本更紧凑）:

```
┌──────────────────────────────────────────┐
│ 每周精选 AI 工具资讯，直达收件箱          │
│ [email input] [订阅]                     │
└──────────────────────────────────────────┘
```

### 周刊入口与展示位置

| 入口 | 位置 | 形式 |
|------|------|------|
| **主导航** | Header | "周刊" 导航项（独立或在"内容" dropdown 下） |
| **首页编辑精选区** | 首页第 3 模块 | 大卡片，展示最新一期 |
| **文章列表页 Tab** | /articles 页面 | "编辑精选" Tab 中置顶展示 |
| **周刊列表页** | /weekly | 独立页面，展示所有往期 |
| **Footer** | 全站底部 | "周刊存档" 链接 |
| **文章详情页** | 侧边栏或底部 | "本文被收录在 Weekly #3" 关联提示 |

### 社交分享触发设计

**当前状态**: 零分享功能。没有任何社交分享按钮。

**推荐分享触点**:

| 位置 | 类型 | 触发时机 |
|------|------|---------|
| 文章详情页标题旁 | 分享按钮组 (Twitter/知乎/复制链接) | 始终可见 |
| 文章详情页阅读完毕 | "觉得有用？分享给同事" + 按钮 | 滚动到底部 |
| 周刊详情页 | "分享本期周刊" 固定按钮 | 始终可见 |
| Skill 详情页 | "推荐这个工具" 按钮 | 安装命令区附近 |
| 文章列表空状态 | (不适用) | - |

**分享文案模板（预填充）**:
- Twitter: `发现一个好用的 AI 工具: [Skill 名称] — via @skillnav_dev [链接]`
- 知乎: `[文章标题] | SkillNav — 中文开发者的 AI 工具生态指南`
- 复制链接: 复制当前 URL + toast 提示"已复制"

**分享的隐性价值**: 每次分享都是免费的外链，对 SEO 有直接帮助。尤其是知乎和掘金上的分享，可以带来高质量的反向链接。

## 6. 关键建议 Top 10

按投入产出比排序（考虑开发成本、对用户体验的影响、对增长的贡献）:

### 1. 文章详情页增加 Newsletter CTA（P0，0.5h）

**现状**: 文章详情页是外部流量的主要着陆页，但底部只有"相关文章"，零订阅入口。
**改动**: 在 `/articles/[slug]/page.tsx` 的 related section 之前插入紧凑版 Newsletter 组件。
**影响**: 直接提升从渠道读者（画像 D）到订阅者的转化率。

### 2. StatsBar 替换为真实数据（P0，0.5h）

**现状**: `stats-bar.tsx` 硬编码了虚假数据（13,000+ Skills 但只有 168 可见，25,000+ MAU 无依据）。这在技术受众面前会严重损害信任。
**改动**: 替换为真实数据（168 精选 Skills / 270+ 翻译资讯 / 13 一手信源 / 每日更新），参考 About 页的 stats 数组（已使用真实数据）。
**影响**: 避免信任崩塌，About 页和首页数据一致性。

### 3. 周刊独立路由 /weekly（P0，2-3h）

**现状**: 内容战略 v2 已设计周刊为品牌核心，但无独立展示路径。
**改动**: 新增 `/weekly` 列表页 + `/weekly/[slug]` 详情页（专属模板：目录导航 + 往期切换 + 分享）。
**影响**: 为周刊发布提供基础设施，周刊上线后立刻可用。

### 4. 导航结构简化为 4 项（P1，1h）

**现状**: "首页 | Skills | MCP | 资讯 | 关于" 共 5 项，首页项多余，Skills/MCP 并列造成困惑。
**改动**: 改为 "探索 | 周刊 | 资讯 | 关于"。"探索"指向 /skills（MCP 入口在 /skills 页面增加 Tab 或在 Footer）。或者保持 "Skills | 周刊 | 资讯 | 关于"，MCP 降级为 Footer 链接。
**影响**: 降低认知负担，突出周刊的品牌地位。

### 5. 首页增加搜索框（P1，1-2h）

**现状**: 首页 Hero 区只有两个 CTA 按钮，没有搜索入口。
**改动**: Hero 区 CTA 下方增加全站搜索框，输入后路由到 /skills?q=xxx（MVP 只搜 Skills）。
**影响**: 对画像 A（效率猎手）直接缩短找工具的路径。

### 6. 文章详情页增加分享按钮（P1，1h）

**现状**: 零分享功能。
**改动**: 文章标题旁增加 Twitter/知乎/复制链接 3 个分享按钮。
**影响**: 每篇文章都变成潜在的分发节点，对 SEO 有长期价值。

### 7. 首页编辑精选区（P1，2h，周刊上线后启用）

**现状**: 首页只有"精选 Skills"和"最新资讯"，无编辑内容展示。
**改动**: 在 FeaturedSkills 上方新增 EditorialHighlights 组件（周刊大卡片 + 评测列表）。editorial 为空时自动隐藏。
**影响**: 首页从"工具导航站"升级为"有编辑立场的策展品牌"。

### 8. Skill 详情页增加关联文章（P2，2h）

**现状**: Skill 详情页只有"相关 Skills"（同分类），无关联文章。
**改动**: 基于 Skill 名称在 articles 表做关键词搜索，展示 2-3 篇相关文章卡片。
**影响**: Skill 页面从"工具卡片"升级为"工具 + 上下文"，用户留存时间提升。

### 9. 文章列表页增加 content_tier Tab（P2，1-2h）

**现状**: /articles 页面所有文章平铺展示。
**改动**: 增加 Tab 切换：编辑精选 / 全部资讯。编辑精选 Tab 只显示 `content_tier = 'editorial'`。
**影响**: 配合内容战略 v2 的双维度内容体系，让用户快速区分编辑内容和翻译内容。

### 10. MCP 数据迁移到数据库（P3，3-5h）

**现状**: MCP 页面使用 `src/data/mcp-servers.ts` 硬编码的静态数据，与 Skills/Articles 的 Supabase 驱动模式不一致。
**改动**: MCP 数据迁入 Supabase，复用 Skills 的 DAL 模式，支持搜索/分类/分页。
**影响**: 为 MCP 数据的增长提供基础，统一技术架构。
