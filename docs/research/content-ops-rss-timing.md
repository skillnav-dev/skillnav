## 核心矛盾

**源发布高峰 (CST 00:00-06:00) ≠ 读者消费高峰 (CST 08:00-10:00 / 20:00-23:00)**

→ 需要在两个消费高峰之前完成采集+翻译

## 同步方案对比

### 当前: 单次同步 UTC 00:00

- CST 08:00 开始采集 → ~08:20 完成 → 读者 08:00 错过当天新内容
- 只能捕获到 UTC 00:00 之前的内容（= 前一天 US 下午发布的）
- 当天 US 发布的内容要等到次日才能同步

### 方案 A: 单次调整至 UTC 22:00 (最小改动)

- CST 06:00 开始 → ~06:20 完成 → 08:00 早高峰前就绪
- 捕获当天 UTC 16:00-22:00 高峰期内容
- 缺点: 仍然是 24h 间隔，欧洲源日间内容要等到次日

### 方案 B: 双同步 UTC 22:00 + UTC 10:00 (推荐)

- **晨间** UTC 22:00 (CST 06:00): 捕获 US 全天发布 → 08:00 早高峰就绪
- **午后** UTC 10:00 (CST 18:00): 捕获欧洲日间 + 补漏 → 20:00 晚高峰就绪
- 12h 间隔，最大时效性
- 额外成本: ~50% GitHub Actions 分钟数（第二次多数 dedup）

### 方案 C: 三同步 UTC 06:00 + UTC 14:00 + UTC 22:00

- 8h 间隔，最大覆盖
- 但收益递减，大部分源每天只发 1-3 篇
- 不推荐: 复杂度和成本不匹配

## 建议

**推荐方案 B（双同步）**，理由:
1. 精准覆盖两个读者高峰
2. 12h 间隔大幅降低"新闻过夜"问题
3. 第二次同步因去重机制，实际 LLM 调用很少
4. GitHub Actions 免费额度 2000 分钟/月，每次 ~16min，双同步 = ~960 min/月，在配额内
