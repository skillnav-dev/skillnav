# SkillNav 论文频道方案 v3

> 状态：done（M1+M2 完成，M3 持续迭代）
> 日期：2026-03-26
> 基于：v2 方案 + 6 agent 并行调研（竞品/技术/内容/分发/商业化/架构）
> 前版：`paper-channel-proposal-v2.md`（Phase 1 已上线，本文为其升级方案）

---

## 0. 一句话

**日报论文板块从"一句话推荐"升级为"300 字导读卡"（每天 3+ 篇），好论文一键触发全文翻译存档。不建新产品线，不加新页面，改一个脚本 + 加一个脚本。**

---

## 1. 为什么现在做（而不是等观察期结束）

v2 设定了 4 周观察期（~04-23），商业化 agent 建议"等数据再说"。但三个新事实改变了判断前提：

| 新事实 | 影响 |
|--------|------|
| HF API 已返回 `ai_summary`、`ai_keywords`、`githubRepo`、`githubStars`，现有代码未使用 | 导读生成的数据层已就绪，不需要下载 PDF，工程成本趋近于零 |
| 竞品调研确认：中文市场没有"开发者视角的论文导读" | 差异化窗口存在，不是在已有赛道里卷 |
| 每天 0-3 篇一句话推荐的信息量不足以验证需求 | 内容太薄，用户点不点都说明不了问题——要给足够好的内容才能测出真实需求 |

**核心逻辑：不是"提前投入"，而是"用几乎零成本把测试品质量拉到能真正验证需求的水平"。**

v2 的纪律仍然有效：不建新页面、不建新表、不做独立产品线。只是把日报里的论文内容从"最小化"升级到"有编辑判断的导读"。

## 2. 产品设计

### 2.1 两层架构：日报导读 + 按需全文

```
                     每日自动
                        |
                        v
HF Daily Papers API ──> generate-daily.mjs ──> 日报
  (Top 10, 含           (LLM 选 3+ 篇,        (300 字导读卡
   ai_summary,           生成导读卡)            + 态度标签
   githubRepo,                                  + 工具关联)
   ai_keywords)
                                                    |
                                          你觉得某篇好
                                                    |
                                                    v
                                          translate-paper.mjs
                                            (ar5iv HTML 优先
                                             PDF fallback
                                             → 全文中译
                                             → 存 articles 表)
```

**设计原则**：自动化处理"发现+初筛+导读"，人工判断只在"要不要全文翻译"这一个决策点。

### 2.2 日报导读卡格式

```markdown
## 论文速递

### 自适应 RAG：让检索只在需要时触发
> Google · NAACL 2026 · 代码已开源 · GitHub 1.2k

**做了什么**：提出轻量分类器判断问题复杂度，简单问题跳过检索直接回答。
三个 QA 数据集上准确率持平，推理开销降 40%。

**对你意味着什么** | 有代码但离生产远
分类器需标注数据训练，不能开箱用。但"先分流再检索"的思路可立刻借鉴
——LangChain 用户看 RouterQueryEngine 会更直观。

**趋势**："不是每个问题都需要检索"正成为 RAG 2.0 共识。

→ [arXiv](tracked-link) · [GitHub](repo-link)
```

**每篇 ~250-300 字，三个板块各有不可替代的功能**：

| 板块 | 功能 | LLM 可自动化程度 |
|------|------|-----------------|
| 做了什么 | 事实层：问题、方法、结果 | 高（摘要改写） |
| 对你意味着什么 | 判断层：能不能用、跟什么工具有关 | 中（需要工具库上下文注入） |
| 趋势 | 视野层：这个方向的温度 | 中（需要近期论文上下文） |

### 2.3 态度标签（四选一，强制）

| 标签 | 含义 | 适用场景 |
|------|------|---------|
| 可以用了 | 有代码+文档，可直接集成 | 开源工具论文 |
| 有代码但离生产远 | 开源了但需大量适配 | 多数研究论文 |
| 纯学术 | 短期无法落地，但方向值得关注 | 理论突破 |
| 思路有启发 | 实验不充分但设计思路好 | 新范式提案 |

**这是竞品没有的东西。** 机器之心告诉你论文说了什么，量子位告诉你这事为什么火，SkillNav 告诉你这论文跟你手里的工具有什么关系、能不能用。

### 2.4 一键全文翻译

触发方式：`node scripts/translate-paper.mjs <arxiv-id>`

```
输入: arxiv ID (如 2603.23483)
  |
  ├─ 1. 获取元数据: arXiv API (标题/作者/摘要/分类)
  |
  ├─ 2. 获取全文 (优先级排序):
  |     a) ar5iv HTML (arxiv.org/html/{id}) → cheerio 提取正文
  |     b) PDF fallback → pdf-parse 提取文本
  |     c) 仅摘要 fallback (如果 a+b 都失败)
  |
  ├─ 3. 分段翻译: 按章节拆分 → DeepSeek 逐段翻译
  |     (保留原文标题/图表引用/公式编号)
  |
  ├─ 4. 存储: INSERT into articles 表
  |     source_url: arxiv 链接
  |     category: 'paper'
  |     tags: ai_keywords from HF API
  |     status: 'draft' (翻译完成后人工审核发布)
  |
  └─ 5. 输出: 翻译完成，文章 ID + 预览链接
```

**成本估算**：一篇 20 页论文 ~14K tokens 输入 + ~8K tokens 输出，DeepSeek 约 $0.03/篇。

### 2.5 质量底线

| 规则 | 触发 | 处理 |
|------|------|------|
| 斧头测试 | "对你意味着什么"换个论文标题还能成立 | 重写或删除该篇 |
| 态度标签缺失 | LLM 未输出四选一标签 | 脚本自动拒绝，不入日报 |
| 论文 ID 编造 | ID 不在 HF API 返回的列表中 | 脚本自动过滤 |
| 摘要太短 | HF `ai_summary` < 100 tokens | 跳过，说明是 demo/note |

## 3. 差异化定位

### 3.1 竞品格局（6 agent 调研结论）

```
深度
  ^
  |  机器之心 (学术翻译, 2000-5000字)
  |  Ahead of AI (月刊深度综述)
  |
  |  The Batch (What/How/Behind 三段式)
  |
  |  SkillNav v3 ← 导读层，开发者视角，工具关联
  |
  |  TLDR AI / SkillNav v2 (一句话推荐)
  |
  |  HF Daily Papers (零解读，纯发现)
  +──────────────────────────────────> 受众
    研究者                          开发者
```

### 3.2 SkillNav 独有的三张牌

| 牌 | 为什么别人打不出 |
|----|---------------|
| **工具关联** | 168 Skills + 4000 MCP 数据库，导读 prompt 可注入相关工具上下文。没有第二个产品同时有论文选稿能力和开发者工具数据库 |
| **态度标签** | 四选一的明确判断，所有竞品都在做"客观摘要"，没人敢说"这论文你别看了" |
| **编辑品牌** | "SkillNav 推荐 = 值得用"，论文导读强化的是同一个品牌承诺，不是第二个承诺 |

### 3.3 明确不做的事

| 不做 | 原因 |
|------|------|
| 长文技术解读 (2000+ 字) | 机器之心的领地，团队产能不支持 |
| 每日 5-10 篇论文 | 量子位模式，需要大编辑团队 |
| 独立 `/papers` 页面 | SEO 价值低 (1.5/5)，AI 搜索正在截流"信息提取型"内容 |
| 学术社区/论文讨论 | PaperWeekly 模式，需要冷启动社区 |
| AI 自动选稿无人审核 | Alpha Signal 模式，失去编辑品牌价值 |

## 4. 技术方案

### 4.1 数据层：HF API 字段补全

`fetchHFDailyPapers` 当前只取 4 个字段，补全到 8 个：

```javascript
// 现在
{ id, title, upvotes, org, url }

// 补全后
{ id, title, upvotes, org, url,
  ai_summary,    // HF 生成的 AI 摘要 (导读输入)
  ai_keywords,   // AI 提取关键词 (工具匹配 + 标签)
  githubRepo,    // GitHub 仓库 (态度标签判断依据)
  githubStars    // Star 数 (成熟度信号)
}
```

### 4.2 Prompt 升级

从"一句话推荐"升级为"300 字导读卡"。关键改动：

1. **输出格式**：从 `{summary, hook}` 改为 `{title_zh, what, implication, attitude, trend, github_url}`
2. **工具上下文注入**：把 `ai_keywords` 在 Skills/MCP 数据库中匹配到的工具名注入 prompt
3. **数量约束**：从"0-3 篇"改为"至少 3 篇，最多 5 篇"
4. **态度标签**：强制四选一
5. **Few-shot 示例**：从 v2 闭环积累的 corrections.jsonl 中提取好/坏示例对

### 4.3 全文翻译脚本

新建 `scripts/translate-paper.mjs`：

```
依赖:
  - cheerio (HTML 解析, 已在项目中)
  - pdf-parse (PDF 文本提取, 新增)
  - scripts/lib/llm.mjs (复用现有 LLM 封装)

流程:
  1. arXiv API 获取元数据
  2. fetch arxiv.org/html/{id} → cheerio 提取 <article> 正文
  3. 如果 HTML 不可用 → 下载 PDF → pdf-parse 提取
  4. 按 <h2>/<section> 分段
  5. 每段独立调 LLM 翻译 (保留术语表一致性)
  6. 拼接 → INSERT articles 表 (status: draft)
```

### 4.4 改动文件清单

| 文件 | 改动类型 | 内容 |
|------|---------|------|
| `scripts/generate-daily.mjs` | 修改 | fetchHFDailyPapers 补字段 + prompt 升级 + 格式编译 |
| `scripts/translate-paper.mjs` | **新建** | 一键全文翻译脚本 |
| `scripts/lib/glossary.json` | 修改 | 补充论文领域术语 |
| `package.json` | 修改 | 新增 `pdf-parse` 依赖 |

**4 个文件，其中 1 个新建，3 个修改。没有 DB 迁移，没有新页面，没有新 API。**

### 4.5 全文翻译存储

复用 `articles` 表，通过字段区分：

```sql
-- 论文全文翻译的文章，用以下字段标识：
source_url LIKE 'https://arxiv.org/%'  -- 来源是 arXiv
-- 可选: tags 包含 'paper-translation'
```

不建新表。articles 表已有 `source_url`、`title`、`title_zh`、`content`、`content_zh`、`tags`、`status` 等字段，完全覆盖论文翻译的需求。

## 5. 与现有体系的关系

### 5.1 日报管线（零冲突）

```
现有流程不变:
  scrape-signals → generate-daily → admin 审核 → publish-daily

v3 改动只在 generate-daily 内部:
  fetchHFDailyPapers 多取 4 个字段
  LLM prompt 输出格式升级
  Markdown 编译逻辑适配新格式
```

### 5.2 文章管线（新增入口）

```
现有入口: sync-articles.mjs (RSS 采集 + 翻译)
新增入口: translate-paper.mjs (手动触发 + 全文翻译)

两个入口写入同一张 articles 表，共享:
  - 质量评分体系 (quality.mjs)
  - 详情页渲染 (/articles/[slug])
  - 管理后台
```

### 5.3 核心飞轮（强化而非分散）

```
资讯翻译(引流) ← 日报导读卡提升洞察力
      |              + 全文翻译扩充内容资产
      v
工具导航(留存) ← 导读卡中的工具关联 → Skills/MCP 页面
      |
      v
Skill 套件(变现) ← 不变
```

论文内容服务于现有飞轮的"引流"环节，不是第二条产品线。

## 6. 成本与风险

### 6.1 运营成本

| 项目 | 日报导读卡 (自动) | 全文翻译 (手动触发) |
|------|-----------------|-------------------|
| LLM API | ~$0.01/天 (3 篇导读) | ~$0.03/篇 |
| 人工 | 日报审核时多看 2-3 分钟 | 翻译后审一遍 ~15 分钟/篇 |
| 基础设施 | 零新增 | pdf-parse 一个依赖 |

**月度总成本**：LLM < $1 + 人工 ~30 分钟/周（假设每周翻译 1-2 篇好论文）

### 6.2 风险

| 风险 | 概率 | 应对 |
|------|------|------|
| 导读卡质量不够，LLM 生成废话 | 中 | 斧头测试 + 态度标签强制 + few-shot 迭代 |
| ar5iv HTML 覆盖率不足 | 低 | PDF fallback；仅摘要 fallback |
| 论文板块篇幅膨胀日报 | 中 | 3 篇 x 300 字 = 900 字，占日报 ~40%。如果过重，精简"做了什么"段到 2 句 |
| 与机器之心/量子位竞争 | 低 | 不在同一维度竞争（他们做深度/新闻，我们做工具关联） |

## 8. 成功指标

### 导读卡（自动）

| 指标 | 衡量方式 | 目标 |
|------|---------|------|
| 论文链接点击率 | Umami `/go/paper/*` | > highlights 点击率的 50% |
| 态度标签准确率 | 人工审核记录 | > 80% 无需修改 |
| 日报整体打开率 | RSS/邮件 | 不下降 |

### 全文翻译（手动触发）

| 指标 | 衡量方式 | 目标 |
|------|---------|------|
| 使用频率 | 你自己用了几次 | > 2 次/周说明有需求 |
| 翻译质量 | 人工审核 draft | 术语一致、结构完整 |
| 文章阅读量 | Umami | 基线对比（与翻译文章同期） |

### 4 周决策门槛

| 信号 | 决策 |
|------|------|
| 导读点击率好 + 全文翻译用得多 | **Go Phase 2**：/papers 页面 + brief_papers 表 + Skill 命令 |
| 导读点击率好 + 全文翻译不怎么用 | **Hold**：保持导读，暂停全文翻译 |
| 导读点击率差 | **Kill 导读，保留全文翻译**（如果翻译对你个人有用的话） |

## 9. 与 v2 的变更对照

| 维度 | v2 | v3 | 变更原因 |
|------|-----|-----|---------|
| 日报论文数 | 0-3 篇 | 3-5 篇 | 用户决定：至少 3 篇 |
| 论文内容深度 | 一句话推荐 | 300 字导读卡 | HF API 未用字段使成本趋零 |
| 态度标签 | Phase 2 预案 | 现在就做 | 零成本，竞品差异化核心 |
| 全文翻译 | 不在计划中 | M2 新增 | 用户需求：好论文要全文保存 |
| 观察期 | 4 周后决策 | 保持，但测试品质量提升 | 用更好的内容测出更真实的需求 |
| 新页面 | Phase 2 预案 | 仍不做 | SEO 价值低，保持纪律 |
| 新数据表 | Phase 2 预案 | 仍不做 | 复用 articles 表 |

**v3 的本质：v2 的纪律 + 更高质量的测试内容 + 全文翻译作为编辑工具。**
