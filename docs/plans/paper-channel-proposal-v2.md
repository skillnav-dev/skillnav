## 评审综述：v1 的核心问题

四个独立评审者从不同视角审查了 v1 方案，达成以下共识：

| 问题 | 严重性 | 四位评审者意见 |
|------|--------|--------------|
| 过度工程化 | **致命** | 全票通过。2-3 个会话的工程量用于验证一个未经证实的假设 |
| 需求未验证 | **致命** | CEO + 魔鬼代言人：没有任何数据证明 SkillNav 用户想看论文 |
| 日报塞 5-10 篇论文 | 严重 | 全票否决。违反"克制即品牌"原则 |
| 导读卡 6 板块过重 | 严重 | 主编 + 用户研究：板块语义重叠，"方法要点"价值最低 |
| 每天 20 分钟审核 | 严重低估 | 用户研究：真正有深度的审核每篇 15-30 分钟 |
| 没有编辑声音定义 | 严重 | 主编："没有就会失败" |
| Skill 作为核心分发 | 高估 | 用户研究：受众 B 不在 IDE 里查论文 |
| TAM 分析脱离实际 | 中 | CEO："50-80万人跟你没有关系" |

**评审们的共同建议：不要建 pipeline，先用最小成本测试需求。**

## 1. 做什么（唯一一件事）

**在现有日报中加一个"论文速递"板块，每天最多 3 篇，4 周后看数据。**

### 1.1 具体操作

`generate-daily.mjs` 的 LLM prompt 增加一条规则：

> 从 HuggingFace Daily Papers 当天热门中选出 0-3 篇与 AI 开发者工具/实践最相关的论文。
> 每篇用一句中文概括（20-30 字）+ 机构标签 + 一句编辑钩子（为什么开发者该关心）。
> 如果今天没有值得推的论文，不出这个板块。"今天没有值得推的论文"本身就是编辑判断。

### 1.2 日报中的呈现

```markdown
## 📄 论文速递

- **把世界模型引入 Agent 推理循环** (DeepMind, ICLR oral)
  对 LangChain Agent 用户有直接启发 → [arXiv](链接)

- **自适应 RAG：按需触发检索** (Google, 代码已开源)
  如果你在用 RetrievalQA，这个方向值得跟 → [arXiv](链接)

- **小模型 Reasoning：2B 模型追上 70B** (Microsoft)
  边缘部署场景的人注意了 → [arXiv](链接)
```

**关键设计**：每条都有一句"编辑钩子"（对 XX 用户有直接影响 / 值得跟 / 注意了），不只是标题 + 摘要。这句话决定了用户是否点击。

### 1.3 数据来源

不建新脚本。两种方式取论文数据：

**方式 A（推荐）**：`generate-daily.mjs` 在生成日报时，增加一步 WebFetch 调用 `https://huggingface.co/api/daily_papers`，取当天 Top 10 论文的标题、upvotes、arXiv URL。作为 context 传给 LLM，让 LLM 从中选 0-3 篇。

**方式 B（更简单）**：LLM 自身知识库中有最近的热门论文信息，直接在 prompt 中要求"如果今日资讯与近期重要论文相关，可引用 arXiv 链接"。零代码改动。

### 1.4 工程量

- 改动文件：`generate-daily.mjs`（prompt 扩展 + 可选 HF API 调用）
- 改动文件：`parse-brief.ts`（支持解析 `## 📄 论文速递` section）
- 不改数据库 schema
- 不建新脚本
- 不建新页面
- 不建新 API

**预计工程量：半天。**

### 1.5 你的日常投入

审日报时多花 2 分钟看论文速递板块是否合理。可以删减、调整钩子、或把整个板块删掉（如果当天论文质量不行）。

不额外增加任何运营负担。

## 3. 成功指标与决策门槛

### 3.1 4 周观察指标

| 指标 | 衡量方式 | 说明 |
|------|---------|------|
| 论文链接点击率 | Umami 事件追踪 | 日报中论文 arXiv 链接的点击数 |
| 论文板块 vs highlights 对比 | Umami | 论文链接点击率 vs highlights 链接点击率 |
| 用户反馈 | 日报回复 / 社交媒体 | 有人提到论文内容吗？ |
| 日报整体打开率变化 | RSS/邮件指标 | 加了论文后打开率是否下降（稀释效应） |

### 3.2 决策门槛

**4 周后，根据数据做三选一：**

| 数据信号 | 决策 |
|---------|------|
| 论文链接点击率 > highlights 的 50% | **Go**：启动 Phase 2（导读产品） |
| 论文链接有人点但低于 highlights | **Hold**：保持日报论文板块，优化编辑钩子，再观察 4 周 |
| 论文链接几乎没人点 / 日报打开率下降 | **Kill**：撤掉论文板块，精力回归核心产品 |

## 4. 如果 Go：Phase 2 的设计方向（预案，不现在执行）

基于四位评审者的共识，Phase 2 如果启动，应与 v1 方案有以下关键差异：

### 4.1 导读卡片精简为 3 板块

v1 的 6 板块（一句话 / 核心贡献 / 方法要点 / 开发者视角 / 趋势信号 / 元数据）被评审者一致批评为过重且语义重叠。

采纳主编建议的结构：

```markdown
# [中文重写标题]
> [原标题] · [机构] · [会议] · Code: [有/无]

## 这篇论文做了什么
[核心问题 + 关键方法（一句话）+ 主要结果 + 关键数据。3-5 句话。]

## 对你意味着什么
[能不能用 + 怎么影响你现在用的工具/框架 + 该不该读原文。2-3 句话。]

## 趋势信号
[这个方向的温度 + 相关论文/工具。1-2 句话。]
```

**目标字数：250-300 字**（v1 的 400-600 字被认为偏长）。三个板块，每个有不重叠的明确功能。

### 4.2 编辑声音定义（Voice Guide）

Phase 2 启动前必须完成。基于主编建议：

**SkillNav 论文声音 = 工程师同事的技术备忘录（tech memo from a smart colleague）**

| 维度 | 要 | 不要 |
|------|------|------|
| 判断 | "这比 LangChain 现有做法好在 X" | "值得关注" |
| 温度 | "终于有人把这个问题做对了" | "重大突破" |
| 词汇 | "推理开销""模型调不动" | "computational overhead""收敛困难" |
| 态度 | "听起来好但 benchmark 太理想化" | 只说好话 |
| 句式 | 短句为主 | 学术论文的从句嵌套 |

**斧头测试**：如果把论文标题换掉，这句编辑点评还能成立，就是废话，不能发布。

### 4.3 LLM 质量控制强化

v1 对 LLM 输出质量预期过于乐观。Phase 2 需要：

- **态度标签强制选择**：prompt 要求 LLM 从 4 个选项中选一个：`可以用了` / `有代码但离生产远` / `纯学术，短期无法落地` / `思路有启发但实验不充分`
- **上下文注入**：把论文关键词在 SkillNav 数据库中搜到的相关 Skill/MCP 列表注入 prompt，让"对你意味着什么"能具体到工具名
- **人工审核只聚焦一个点**：不审 3 个板块，只审"对你意味着什么"这一句。如果这句话你写不出来，这篇论文就不该做导读

### 4.4 导读数量控制

不是每天 2-3 篇。**每周 3-5 篇精选导读**，质量优先于数量。每篇实际人工投入 15-20 分钟（包括读论文核心部分 + 重写"对你意味着什么"）。

### 4.5 Skill 命令（优先级降低）

`/skillnav papers <keyword>` 作为 Phase 2 的可选项，不是必做项。搜索是低频场景，浏览（日报/网站精选列表）才是主路径。如果做，可以考虑魔鬼代言人的建议——直接查 Semantic Scholar API，不需要自己建论文数据库。

## 5. 战略定位的修正

### 5.1 论文不是新产品线，是编辑素材

v1 把论文当成一个独立频道来设计（新页面、新 API、新 Skill 命令）。评审后的认知修正：

**论文内容的价值不在于它作为独立产品的留存力，而在于它作为编辑素材让日报变得更有洞察力。**

用户的 aha moment 不是"SkillNav 有论文导读页面"，而是"SkillNav 的日报能告诉我技术方向在往哪走"。这个洞察来自用户研究评审。

### 5.2 不做第二个品牌承诺

v1 说"从工具导航升级为工具导航 + 技术前哨"。CEO 评审指出：SkillNav 连"工具导航"这个认知都还没建立，不该同时做两个承诺。

**v2 定位**：论文内容是工具导航的深度补充，不是独立的第二条腿。"SkillNav 推荐的工具，背后的技术趋势我们也帮你看了"——这仍然是一个品牌承诺（编辑判断力），不是两个。

### 5.3 诚实面对团队产能

方案中的每一行代码、每一条 pipeline 都是未来的维护成本。pipeline-reliability 项目刚完成（D1-D3），那次经验告诉我们：**每一条新 pipeline 都是新的故障面**。

在 Phase 1（日报板块测试）阶段，不增加任何运维负担。只有数据验证通过后，才考虑增加基础设施。

## 6. 关于投资者受众

v1 的决策"短期不做投资者"获得所有评审者认可。

补充一点：即使未来做，也不是"论文 → 投资解读"，而是"趋势报告 → 决策支持"。这属于变现产品（月度/季度趋势报告），不属于论文频道。两者不应混为一谈。

## 8. 持续优化闭环

> 基于 3 agent 专项调研：内容产品反馈闭环最佳实践（Morning Brew/TLDR/Hustle）、SkillNav 现有质量体系代码级分析、独立运营者可持续性研究（Matt Levine/Lenny Rachitsky）。

### 8.1 核心理念

**你每天审日报时做的每一次修改，都是最高质量的训练信号。大多数产品把这些纠正扔掉了——我们要把它变成系统进化的燃料。**

这不是一个额外的工作流，而是你现有审日报动作的自然延伸。关键洞察来自三个方向：

- **TLDR（50万+订阅）的选稿标准**：只有一个测试——"读到这篇，我会不会转发给朋友？"不需要复杂打分，一个直觉判断就够
- **The Hustle 的质量指标**：用 CTR（点击率）而非打开率做判断。打开不代表内容好，点击才是
- **Matt Levine（Money Stuff，日更 10+ 年）的防倦怠方法**：固定结构降低每天的决策成本，只写自己觉得有意思的东西

### 8.2 三层节奏：日记录 → 周聚类 → 月进化

整个闭环围绕一个简单的原则运转：**先记录，后分析，再改进。不要同时做。**

```
┌─ 每日：记录（2 分钟）─────────────────────────────┐
│                                                    │
│  审日报时，遇到问题随手记一笔：                       │
│                                                    │
│  corrections.jsonl:                                │
│  {"date":"03-26", "type":"hook", "before":"值得关   │
│   注", "after":"如果你在用 LangChain 的 RAG，这个   │
│   方向值得跟", "paper":"adaptive-rag"}              │
│                                                    │
│  五种纠正类型（只需选一个标签）：                      │
│  · hook   — 编辑钩子太泛/不准/无吸引力              │
│  · term   — 术语翻译不对或不一致                    │
│  · miss   — 遗漏了应该出现的论文/方向               │
│  · noise  — 选进了不该出现的论文                    │
│  · tone   — 语气太学术/太随意/不像 SkillNav         │
│                                                    │
│  不需要分析，不需要改 prompt，就是记下来。             │
└──────────────────────┬─────────────────────────────┘
                       │
                       ▼
┌─ 每周一：聚类 + 微调（15 分钟）────────────────────┐
│                                                    │
│  1. 扫一遍本周的 corrections.jsonl（2 分钟）        │
│     哪种类型最多？hook？term？miss？                │
│                                                    │
│  2. 最高频的类型 → 改一个东西（10 分钟）             │
│     · hook 问题多 → 在 prompt 里加 2 个好/坏示例对  │
│     · term 问题多 → 更新 glossary.json              │
│     · miss 问题多 → 检查 HF API 参数或加关键词      │
│     · noise 问题多 → 加负面筛选条件到 prompt        │
│     · tone 问题多 → 开始积累 voice guide 素材       │
│                                                    │
│  3. 看 Umami 论文链接点击数据（3 分钟）              │
│     哪些论文被点了？哪些没人点？                      │
│     点击高的论文有什么共性？                          │
│                                                    │
│  规则：一次只改一个变量。改三个你不知道哪个有效。      │
│  规则：每次改动记版本号和原因（prompt 头部注释）。     │
└──────────────────────┬─────────────────────────────┘
                       │
                       ▼
┌─ 每月最后一个周一：进化（30 分钟）─────────────────┐
│                                                    │
│  1. 源质量评分（10 分钟）                           │
│     每个论文来源的"命中率"= 选入日报的论文数 /       │
│     该源总采集数。命中率 < 20% 的源考虑降权或移除。   │
│     新增的源先观察一个月再评。                        │
│                                                    │
│  2. 术语表审查（10 分钟）                           │
│     本月新增了哪些术语？标记了 confidence: low 的     │
│     词是否已经稳定？可以提升为 high 了？              │
│     有没有官方推荐译名更新需要同步？                  │
│                                                    │
│  3. Prompt 版本回顾（10 分钟）                      │
│     本月改了几版 prompt？效果如何？                   │
│     有没有改了反而变差的？（需要回滚）                │
│     corrections.jsonl 中某类问题是否明显减少了？     │
│                                                    │
│  产出：更新一句话到本方案末尾的"闭环演进记录"        │
└────────────────────────────────────────────────────┘
```

**时间总预算**：每天 2 分钟 + 每周 15 分钟 + 每月 30 分钟 = **每月约 3 小时**。这是可持续的。

### 8.3 四个具体改进通道

#### 通道一：编辑钩子（hook）质量

论文速递的"为什么开发者该关心"那句话是点击率的决定因素（The Hustle 验证：好标题多拿 43,876 次点击）。

**改进方式**：Prompt 中积累 few-shot 示例对。

```
好的 hook：
  "如果你在用 LangChain 的 RAG，这个方向值得跟"
  "边缘部署场景的人注意了"
  "终于有人把 Agent Memory 做对了"

坏的 hook（斧头测试：换个论文标题还能成立 = 废话）：
  "值得关注"
  "对开发者有启发"
  "这个方向很热门"
```

每周从 corrections.jsonl 中提取好/坏 hook 对，加进 prompt。4 周后你会有 8-12 个示例对——这已经足够让 LLM 学会你的钩子风格。

#### 通道二：术语表（glossary）演进

现有 glossary.json 有 57 条术语（30 keep + 9 translate + 18 bracket）。论文领域会带来新术语。

**演进规则**（借鉴 CSET-ETO 中文技术术语表模式——一人维护，按需更新）：

- **翻译中遇到新词 → 当场加进 glossary.json**，标记 `confidence: low`
- **月度审查时**：low confidence 词用了一个月没问题 → 提升为 high
- **读者/自己发现不一致**：立刻修正，记录变更原因

不需要额外工具，glossary.json 本身就是知识库。

#### 通道三：源质量评分

借鉴情报分析的 Admiralty Code（NATO 用了 80+ 年），简化为一个指标：

**命中率 = 该源论文被选入日报的比例**

```
Phase 1 的源只有两个，简单记录即可：
  HF Daily Papers: 命中率 = 3/10 = 30%（每天 Top 10 中选 3 篇）
  [未来] arXiv 类目: 命中率待观察

命中率持续 < 10% 的源 → 降权（LLM 选稿时降低优先级）
命中率持续 > 50% 的源 → 升权（优先展示）
```

Phase 1 阶段只有 HF 一个源，命中率自然就是"10 篇里选几篇"。等 Phase 2 加更多源时，这个评分机制就会变得关键。

#### 通道四：编辑声音的自然生长

不预设 voice guide。**让 voice guide 从每日纠正中自然长出来。**

> "一开始我拼命写笑话，硬凑'互联网风格'。随着时间推移，很多东西自然脱落了，留下来的更像是我自己的思考节奏。" —— Matt Levine

**操作方式**：corrections.jsonl 中 `type: "tone"` 的纠正积累 4 周后，你会发现自己反复做同样的修改——那些修改就是你的 voice。提取出来，就是 voice guide 的初稿。

Phase 2 启动前需要的 voice guide 不是凭空设计的，而是从 4 周日记录中**提炼**出来的。

### 8.4 与现有质量体系的对齐

现有文章管线有三层质量门控（ADR-004），论文闭环复用同样的思路，但 Phase 1 更轻：

| 层级 | 现有文章管线 | 论文管线 Phase 1 | 论文管线 Phase 2（远期） |
|------|------------|-----------------|----------------------|
| L0 配置 | 源关键词白名单（拦截 74%） | HF API Top 10 过滤 | + arXiv 类目 + 源质量评分 |
| L1 规则 | content_zh < 200字 → draft | LLM prompt 筛选规则 | + 负面示例 + few-shot |
| L2 判断 | LLM audience_fit + credibility | **你本人**（日审） | LLM 初筛 + 你终审 |

**关键发现**：现有 L2 评分（audience_fit / credibility / reason）只 log 到控制台，没有持久化到数据库。这意味着我们无法回溯分析历史质量趋势。建议 Phase 2 时把 L2 评分存进 articles 表——对文章管线和论文管线都有价值。

### 8.5 渐进自动化路线图

不要一步到位。**先手动做，发现模式，编码为规则，然后自动化。**

```
第 1-2 周：全手动
  · 每天扫 HF Daily Papers，手动挑 3 篇，手写 hook
  · 把每次判断记进 corrections.jsonl
  · 目的：建立直觉，发现"我到底怎么选论文"的内隐规则

第 3-4 周：半自动
  · LLM 从 HF Top 10 中推荐 3 篇（prompt 含 2 周积累的 few-shot）
  · 你审核：接受/替换/修改 hook
  · 把纠正继续记录，对比 LLM 推荐 vs 你的选择

第 5-8 周：看数据做决策
  · Umami 点击率数据 + corrections 聚类分析
  · 决策：Go（Phase 2） / Hold / Kill
  · 如果 Go：4 周的 corrections 直接转化为 voice guide + prompt

第 9+ 周（仅 Phase 2）：
  · 导读生成 prompt 含完整 few-shot + voice guide
  · 源质量评分自动计算
  · 你只审核"对你意味着什么"一句话
```

### 8.6 可持续性保障

**编辑时间预算硬限**：

| 活动 | 时间上限 | 频率 | 超时处理 |
|------|---------|------|---------|
| 日报审核（含论文） | 7 分钟 | 每天 | 超时就少改几条，不影响发布 |
| 纠正记录 | 2 分钟 | 每天 | 来不及就跳过，下次记 |
| 周度聚类 + 微调 | 15 分钟 | 每周一 | 硬停，未完成的下周继续 |
| 月度进化 | 30 分钟 | 每月最后周一 | 硬停，只做最重要的 1 件 |

**防倦怠原则**（来自独立运营者研究）：

- **没有好论文的日子，就不出论文板块**。"今天没有值得推的论文"本身就是编辑判断，不是失职
- **先建立习惯，再增加复杂度**。第一周只做"记录纠正"一件事，稳定后再加周度聚类
- **质量地板，不是质量天花板**。"宁漏发不误发"已经是最好的质量标准，不需要更多

## 9. 实施计划

### 现在做（半天工程量）

1. `generate-daily.mjs`：LLM prompt 增加论文速递规则 + HF Daily Papers API 调用
2. `parse-brief.ts`：支持解析 `## 📄 论文速递` section

4. 开始 4 周观察期
5. 每天审日报时按第 8 节的反馈机制记录问题，持续调优 prompt 和筛选逻辑

### 4 周后决策

基于数据 + 4 周积累的反馈经验：

- **Go**：启动 Phase 2（导读产品），4 周的反馈直接转化为 voice guide 和 prompt 质量
- **Hold**：保持板块，继续优化（反馈积累越多，质量越好）
- **Kill**：撤掉板块，反馈经验仍可用于提升现有日报质量

- Voice guide 编写（半天）
- `generate-paper-cards.mjs`：LLM 生成 3 板块精简导读
- `/papers` 列表页 + `/papers/[slug]` 详情页
- DB：articles 表新增 paper_meta JSONB
