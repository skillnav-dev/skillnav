第一章

什么是 RAG

Retrieval-Augmented Generation — 检索增强生成

大模型的两个核心痛点

大语言模型虽然强大，但存在两个根本性的局限：知识截止（训练数据有时间限制，不知道最新信息）和幻觉（当不确定时可能自信地编造答案）。RAG 正是为了解决这两个问题而设计的。

RAG 与普通 LLM 对比

❌ 无 RAG

模型仅依赖训练数据，无法访问私有知识，可能产生幻觉，知识有截止日期。

易产生幻觉 知识过时 无法个性化

✅ 有 RAG

模型先检索再回答，能引用真实来源，知识实时更新，答案可追溯验证。

有据可查 知识实时 减少幻觉
一句话总结：RAG 的核心思路就是三个字——查→拼→答。先从外部知识源中检索相关内容，再把这些内容塞进 Prompt 里，最后让模型基于真实材料来回答。
第二章

优化策略全景

检索前 · 检索中 · 检索后 — 八大核心策略

检索前是在"备好弹药"——构建高质量的向量空间。

Chunking 文档分块

›
把长文档切成合适大小的片段。固定长度切分最简单但粗暴；语义切分根据主题边界切割；递归切分先按段落、再按句子逐层细分。通常会加 10-20% 的重叠（overlap）来保留上下文连续性。

Embedding 模型选择

›
Embedding 模型决定了"语义空间"的质量。需要考虑维度大小（越高越精确但越慢）、多语言支持、领域适配性。可以在通用模型基础上用领域数据做 fine-tune。

Metadata enrichment 元数据增强

›
给每个 chunk 附加来源、日期、类别、作者等元数据，检索时先用元数据过滤，缩小搜索范围后再做向量匹配。

检索中是在"打得更准"——混合检索与查询改写。

Hybrid search 混合检索

›
纯向量搜索擅长理解语义（"退货"≈"退换货"），但可能漏掉精确关键词匹配（如产品编号 SKU-1234）。纯关键词搜索（BM25）擅长精确匹配但不懂同义词。两者结合效果最好。

Query transformation 查询改写

›
Query expansion — 用 LLM 生成同义改写和多角度子问题。
HyDE — 让 LLM 先"假设性回答"，再用假设答案去检索。
Step-back prompting — 把具体问题抽象为更宽泛的问题。

检索后是在"精选精炼"——重排序与自适应策略。

Re-ranking 重排序

›
初次检索用轻量级 bi-encoder，速度快但精度有限。Re-ranker 用 cross-encoder 对 query 和每个候选文档做精细打分，把最相关的排到最前面。

Context compression 上下文压缩

›
检索到的 chunk 中可能只有一两句话真正相关。压缩器抽取每个 chunk 中与 query 相关的句子，减少 token 数量，降低成本且提升精度。

Self-RAG / CRAG 自适应检索

›
让模型自己判断是否需要检索、结果是否有用。Self-RAG 让模型在生成过程中动态决策；CRAG 在检索后加一步"质量评估"，不合格的结果会被丢弃或触发网络搜索补充。
第三章

Chunking 与 Overlap

文档分块策略与重叠比例的场景适配

为什么 Overlap 很重要

Overlap 是相邻两个 chunk 之间的"缓冲带"。它防止一个完整的语义单元（一句话、一个论点、一段代码逻辑）被切断在边界上，导致两个 chunk 各拿到一半。

高重叠
经验法则：overlap 通常设在 chunk 大小的 10%–25% 之间。低于 10% 容易丢失边界上下文，高于 25% 则存储膨胀明显且收益递减。先从 15% 开始实验，根据检索召回率调整。
第四章

HyDE — 假设文档嵌入

Hypothetical Document Embeddings：用假设性回答提升检索精度

核心洞察

用户的问题和知识库的文档"长得不像"，但 LLM 生成的假设回答和知识库文档"长得很像"。传统 RAG 是拿一句短问题去匹配一段长文档（"点 vs 区域"），HyDE 变成了"文档 vs 文档"的匹配，精度自然更高。

HyDE 四步工作流

1. 用户提问

"量子计算对密码学有什么影响？"

2. LLM 生成假设性文档

"量子计算机利用 Shor 算法可以在多项式时间内分解大整数，这将使 RSA 和 ECC 面临根本性威胁。后量子密码学正在研究基于格的密码……"

⚠ 这段回答可能不准确，但我们只需要它的"语义方向"

3. 用假设文档的向量去检索

假设文档和真实文档形态相似——都是长段落、都包含专业术语。"文档 vs 文档" 的匹配更精确。

4. 用真实文档生成最终回答

丢弃假设文档 → 用检索到的真实文档 + 原始问题 → LLM 生成有据可查的回答。

类比

想在图书馆找一本书。传统方法是拿着书名去搜索；HyDE 是先自己写一段"这本书大概讲什么"，然后拿这段描述去匹配——因为你的描述和真实书的摘要更像。

优缺点

优点

  • 零训练成本，即插即用
  • 显著提升语义模糊查询的召回率
  • 跨语言效果好
  • 与任何 embedding 模型兼容

缺点

  • 多一次 LLM 调用，延迟 +1-3 秒
  • LLM 生成方向偏离时会"带歪"检索
  • 对精确关键词查询无优势
  • API 成本翻倍
适用场景：探索性问题、概念性查询、跨语言检索、用户提问模糊宽泛的场景。
不适合：精确查找（订单号、SKU）、低延迟要求、LLM 对领域完全陌生的场景。
第五章

Self-RAG & CRAG

从固定流水线到自适应智能体的进化

核心机制：4 种反思 Token

Retrieve 需要检索吗？ IsRel 文档相关吗？ IsSup 生成有依据吗？ IsUse 回答有用吗？

接收问题 → 判断是否检索

模型输出 Retrieve=yes/no，如果 no 则直接用自身知识回答。

并行生成 + 三维评分

对每个检索文档并行生成候选回答，同时输出 IsRel / IsSup / IsUse 三个维度的评分。

选择最优候选

综合三个评分，用 beam search 选出得分最高的候选。所有候选分数低时可重新检索。

训练方法详解

›
第 1 步：用 GPT-4 等教师模型对 15-20K 条数据标注四种反思 token 的值。
第 2 步：把反思 token 作为特殊 token 加入词汇表，对 Llama 等基座模型做全量 SFT，用标准 next-token-prediction loss。
第 3 步：推理时可调反思 token 权重来控制行为——想要更多引用就提高 IsSup 权重。

核心机制：检索评估器 + 三条路径

常规检索

和普通 RAG 一样，先检索 Top-K 文档。

评估器打分 → 分三档

T5-large 微调的评估器对每个结果打分。

Correct → 精炼后直接使用

至少有一个文档高度相关，用知识精炼器提取关键信息。

Incorrect → 丢弃 + Web Search

所有文档不相关，触发网络搜索兜底。

Ambiguous → 两路合并

保留精炼后的检索结果和搜索结果，合并送入 LLM。

训练与集成方式

›
评估器：基于 T5-large 微调，训练数据从 QA 数据集中标注 correct/incorrect/ambiguous，几千到一万条即可。
精炼器：无需训练，把文档分解成细粒度"知识条"，评估器逐条打分只保留高分条目。
集成：即插即用中间层，不修改 LLM，任何 RAG 系统都可直接加入 CRAG 模块。

Self-RAG vs CRAG 对比

维度Self-RAGCRAG
核心思路模型内化反思能力外挂质检中间层
是否改 LLM是，需要微调基座否，LLM 不动
训练成本较高：全量 SFT较低：微调小型评估器
决策粒度token 级别query 级别
纠错能力并行候选 + 评分选择web search 兜底
适用场景高质量要求（研究、写作）快速集成的生产系统
简单记忆：Self-RAG 改的是"大脑"（LLM 本身），CRAG 改的是"流水线"（RAG 管道）。前者上限更高但成本也高，后者务实易落地。两者也可以组合使用。
第六章

技术栈与实战

从零搭建 RAG 系统的框架选型与架构建议

核心组件选型

向量数据库

Milvus / Qdrant / Pinecone

Milvus 开源高性能，Qdrant Rust 实现速度快，Pinecone 全托管免运维。小规模可用 Chroma 或 pgvector。

Embedding 模型

BGE / E5 / OpenAI

开源首选 BGE-M3（多语言）或 E5-Mistral；商用首选 OpenAI text-embedding-3 或 Cohere embed-v3。

编排框架

LangChain / LlamaIndex

LangChain 生态大适合通用场景；LlamaIndex 专注 RAG 更深入；轻量级可用 Haystack。

Re-ranking

Cohere Rerank / BGE-Reranker

商用首选 Cohere Rerank API；开源用 bge-reranker-v2 或 cross-encoder/ms-marco。

LLM

Claude / GPT-4 / 开源

商用首选 Claude 或 GPT-4；开源用 Llama 3、Qwen 2.5、Mistral。按成本和质量需求选择。

文档处理

Unstructured / Docling

Unstructured 支持 PDF、Word、HTML 等多格式解析；Docling 专注高质量文档理解。

推荐架构

文档解析
→
Chunking + Embedding
→
向量数据库
→
检索 + Re-rank
→
LLM 生成

分阶段实施建议

第一阶段：MVP（1-2 周）

Chroma/pgvector + LangChain + OpenAI Embedding + Claude/GPT-4。固定 chunk 大小，无 re-ranking。先跑通端到端流程。

第二阶段：优化（2-4 周）

引入混合检索（BM25 + 向量）、添加 re-ranking、优化 chunking 策略（语义切分 + 合理 overlap）、添加元数据过滤。

第三阶段：进阶（4-8 周）

实现 query 改写（HyDE/多路查询）、引入 CRAG 评估模块、添加用户反馈循环、构建评估体系（Ragas/TruLens）。

第四阶段：生产级（8+ 周）

迁移到生产级向量数据库（Milvus/Qdrant）、权限控制、日志监控、A/B 测试框架、自动化数据更新流水线。

评估体系

检索质量指标

  • Recall@K — Top-K 中包含正确文档的比例
  • MRR — 正确文档排名的倒数均值
  • NDCG — 考虑排名位置的归一化增益

生成质量指标

  • Faithfulness — 回答是否忠于参考文档
  • Relevance — 回答是否切题
  • Harmfulness — 是否包含有害信息
最重要的建议：不要试图一步到位。先用最简单的方案跑通端到端，然后用真实查询做评估，找到瓶颈在哪，再有针对性地优化。RAG 系统的好坏 80% 取决于数据质量和 chunking 策略，而非框架选型。
